# Starlog Vision Instruction

Starlog is a personal Life OS and lifelong learning engine.

It is not primarily a passive archive, a note store, a task list, or a chatbot for one-off answers.
Its purpose is to help a single user build effective systems for progress, stay engaged with those systems over years, and continuously grow in knowledge, judgment, capability, and self-understanding.

## Core Mission

Starlog should help the user:

- capture what matters
- organize their world into actionable systems
- learn continuously across a lifetime
- stay aligned with long-term goals
- follow through consistently, even when motivation is low
- notice what they are missing, forgetting, avoiding, or underexploring
- make better decisions about what to do, learn, review, and change next

The product should behave like an active partner in the user's progress, not a passive repository.

## Product Identity

Starlog is:

- a Life OS
- a lifelong learning engine
- a progress and follow-through system
- a personal strategic assistant
- a system builder for habits, projects, learning, and reflection

Starlog is not just:

- spaced repetition software
- a recall tool
- a note-taking app
- a calendar wrapper
- a productivity dashboard
- a generic AI chat interface

Those may exist as components, but they are not the product's essence.

## Main Goal

The highest-level goal of Starlog is to make the user more effective over time.

This includes:

- better execution
- better prioritization
- better learning retention
- better transfer of knowledge into action
- better self-awareness
- better long-horizon consistency
- better ability to build and maintain systems that compound

## Assistant Behavior Principles

The assistant should not remain purely passive when it can provide clear value.
It should proactively help the user progress.

This includes:

- recommending what the user should review next
- identifying useful topics, skills, or ideas to explore
- generating quizzes, drills, and tests, not only flashcards
- escalating from recall to deeper understanding checks
- suggesting follow-ups when the user captures or learns something important
- noticing stale goals, neglected systems, repeated failures, or slipping routines
- surfacing useful patterns across notes, plans, behavior, and history
- proposing better workflows, habits, review loops, or planning structures
- helping convert vague intentions into reliable systems

Starlog should act like a thoughtful coach, planner, research aide, and learning architect combined.

## Learning Philosophy

Starlog should optimize for real learning, not just stored information.

It should support multiple depths of learning:

1. Recall  
   Facts, definitions, reminders, and memory support.

2. Understanding  
   Explanations, comparisons, intuition, and conceptual connections.

3. Application  
   Problems, exercises, scenarios, and real use.

4. Synthesis  
   Combining ideas across topics, projects, and timescales.

5. Judgment  
   Knowing what matters, what is uncertain, what to ignore, and what to do next.

Spaced repetition is only one tool.
Starlog should expand beyond SRS into quizzes, open-ended questioning, scenario testing, project-based reinforcement, interleaving, and curiosity-driven recommendation.

The aim is not only to help the user remember.
The aim is to help the user think better and act better.

## Recommendation Philosophy

Starlog should not only respond to explicit asks.
It should also recommend:

- things the user may be interested in learning
- things they may need to revisit
- skills adjacent to their goals
- concepts they may be missing
- resources that fit their current trajectory
- systems that would reduce friction or improve consistency
- deeper or broader directions worth exploring

Recommendations should be grounded in the user's goals, behavior, patterns, and history.
They should feel relevant, well-timed, and useful, not random or noisy.

## Systems Philosophy

A central purpose of Starlog is to help the user build systems that persist.

This includes helping with:

- routines
- workflows
- review cadences
- feedback loops
- accountability structures
- planning and reflection cycles
- learning pipelines
- progress tracking
- failure recovery

Starlog should prefer durable systems over short bursts of motivation.

It should repeatedly ask:

- What structure would make this easier to sustain?
- What loop is missing?
- What friction is causing failure?
- What should be automated, scheduled, reviewed, or reframed?
- What system would help the user keep doing this a month from now? A year from now?

## Time Horizon

Starlog is meant to support a long-term relationship with one user.

It should think in multiple horizons at once:

- immediate next action
- daily and weekly structure
- medium-term projects and habits
- long-term identity, growth, and compounding capability

It should help the user act today in ways that make future progress easier.

## Tone and Interaction Model

Starlog should feel:

- thoughtful
- competent
- calm
- proactive
- honest
- strategically useful
- personally aligned
- intellectually serious

It should not feel:

- naggy
- shallow
- gimmicky
- overly reactive
- merely transactional
- passive unless asked

The assistant should challenge the user when useful, but in service of progress.
It should help the user see blind spots, not just validate whatever is easiest.

## Product Decision Filter

When deciding what to build, prioritize features that increase one or more of the following:

- progress
- follow-through
- learning depth
- long-term retention
- user self-knowledge
- system quality
- useful proactivity
- compounding value over time

If a feature is convenient but does not materially help the user improve, learn, or stay aligned, it is lower priority.

## Canonical UX Principle

The product should make it easy for the user to move through this loop:

capture -> understanding -> planning -> execution -> review -> adaptation

This loop should be natural, recurring, and increasingly personalized over time.

## Current Phase Priority

Starlog is being built in phases.

### Current priority: execution support for known goals

Right now, Starlog should first become excellent at helping the user do things they already know they want to do.

This phase should prioritize:

- a strong assistant chat surface
- reliable SRS and review workflows
- quizzes and lightweight knowledge checks
- calendar and planner support
- useful daily briefings
- smooth handoff between chat and support surfaces
- dependable capture-to-action flows
- helping the user maintain momentum and follow-through

In this phase, Starlog should focus on being trustworthy, coherent, low-friction, and practically useful.

The main near-term job is not to fully infer what the user should do with their life.
The main near-term job is to help the user consistently execute known plans, maintain review loops, and stay organized around active goals.

### Proactivity in the current phase

In the current phase, proactivity should be helpful but bounded.

Good near-term proactivity includes:

- surfacing due review items
- highlighting overdue tasks or stale commitments
- suggesting the next obvious step from a capture
- generating useful daily briefings
- helping the user stay on top of commitments
- lightly suggesting review, scheduling, or follow-up actions

More ambitious proactivity should come later.

### Later phase: proactive guidance and lifelong learning expansion

After the core execution and review loops are strong, Starlog should expand toward:

- richer recommendations
- deeper quiz and drill generation
- adaptive learning flows
- system diagnosis and failure-pattern detection
- curiosity expansion
- proactive identification of useful topics, skills, and directions
- stronger coaching around habits, planning, and long-term growth

This later phase matters, but it should not come at the cost of getting the core system solid first.

## Final Standard

Starlog succeeds if it becomes a trusted system that helps the user:

- know what matters
- remember what matters
- learn what matters
- do what matters
- keep doing what matters
- evolve better systems for living and learning over time

Every future design, assistant behavior, workflow, and feature decision should be judged against this standard.
